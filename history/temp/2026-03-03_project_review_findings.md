- 证据来源：静态代码阅读 + 本地命令行复现（含单帧场景快速验证）。

  1. 输入文件发现机制依赖当前工作目录
  2. water 主入口配置能力不一致（与金属体系绑定）

## 问题清单（按优先级）

### P0: 单帧数据下电极电位流程会崩溃

- 位置：`src/md_analysis/potential/CenterPotential.py`，`electrode_potential_analysis()`
- 触发条件：`center_potential.csv` 或 `fermi_energy.csv` 仅有 1 行数据（例如单帧/强抽帧）
- 现象：`TypeError: iteration over a 0-d array`
- 根因：
  - 使用 `np.genfromtxt(..., names=True)` 读取单行结构化 CSV 时，可能返回 0 维数组；
  - 后续代码默认其可迭代，直接 `for row in center_data`。
- 影响：
  - 直接中断 `U vs SHE` 分析；
  - 单帧调试、快速验证、稀疏取帧场景不可用。
- 建议修复：
  - 对 `genfromtxt` 返回值做 `np.atleast_1d(...)` 归一化；
  - 补充单帧回归测试，防止复发。

### P1: CLI 对 `md.out` 缺失采用“静默降级”，容易误导

- 位置：`src/md_analysis/CLI.py`，`_cmd_potential()` / `_cmd_all()`
- 触发条件：用户传入的 `--md-out` 路径不存在
- 现象：
  - 代码将 `md_out_path` 置为 `None` 并继续执行；
  - 用户可能以为完成了完整流程，实际上部分结果（如 `U vs SHE`、厚度敏感性）被跳过。
- 影响：
  - 结果完整性不透明；
  - 易出现“命令成功但结论不完整”的使用风险。
- 建议修复：
  - 默认 fail-fast（参数错误即退出）；
  - 或至少打印显式 warning，明确哪些子流程被跳过、为何跳过。

### P2: 单位与文案存在不一致，影响结果解释

- 位置：
  - `README.md`（orientation-weighted density 单位写作 `1/Å³`）
  - `src/md_analysis/water/Water.py`（图轴标签出现 `g·cm^3` 文字）
  - 多处 CSV/header/docstring 使用 `g/cm³` 语义
- 问题：
  - 同一分析量在文档、CSV、绘图文本中单位口径不一致；
  - 绘图标签有明显排版/指数错误（应为体密度而非体积量纲）。
- 影响：
  - 对外展示与论文/报告复用时容易引发误读；
  - 下游脚本自动解析单位时可能产生歧义。
- 建议修复：
  - 统一口径为“质量密度加权项”的单位表达；
  - 在 README 与图轴标签同步修正；
  - 在 `history/architecture/modules/data_contract.md` 增补该量的最终单位定义。

### P2: 测试覆盖偏向 happy path，关键边界缺少守护

- 现状：
  - 当前测试可通过（`34 passed`），但覆盖集中在典型流程；
  - 未见针对 CLI 参数错误行为、单帧合并、静默降级提示等边界场景的测试。
- 影响：
  - 线上或复现实验时，最容易踩的边界问题难以及时发现；
  - 重构时容易回归到“看似通过但场景缺失”的状态。
- 建议修复：
  - 增补以下最小集测试：
    - 单帧 `electrode_potential_analysis` 回归测试；
    - `--md-out` 缺失时的 CLI 行为测试（报错或告警）；
    - `--frame-start/end/step` 异常组合的参数校验测试。

## 建议执行顺序（低成本优先）

1. 先修 P0（小改动，高收益，且可直接补回归测试）。
2. 再修 P1（明确行为策略：fail-fast 或 warning）。
3. 同步修 P2 文案与单位（README + 图轴 + 数据契约）。
4. 最后补测试矩阵，形成长期防回归保护。
